# 아키텍처 안티패턴

> 개인 프로젝트에서 반복적으로 나타나는 6가지 설계 실수. 에이전트는 설계 제안 시 이 안티패턴에 해당하는 요소가 없는지 반드시 점검한다.

---

## Overview

아키텍처 안티패턴은 단기적으로는 문제 없어 보이지만, 시간이 지날수록 시스템의 유지보수성·확장성·이해 가능성을 심각하게 저하시키는 설계 패턴이다. 에이전트는 [[10-project-classification]] → [[11-architecture-decision-guide]] 과정에서 이 목록을 체크리스트로 사용한다.

---

## Key Concepts

### 1. Big Ball of Mud (스파게티 아키텍처)

**정의**: 시스템에 명확한 구조나 경계가 없이, 모든 컴포넌트가 서로 뒤엉켜 있는 상태.

**발생 원인**:
- 처음부터 모듈/계층 경계를 정하지 않고 기능 추가
- 빠른 출시를 우선시하며 리팩토링을 계속 미룸
- 코드 리뷰/설계 가이드라인 부재

**신호 (Signs)**:
```
❗ 한 파일/모듈을 수정했는데 전혀 관련 없는 기능에서 버그 발생
❗ 특정 기능이 어디서 처리되는지 찾기 위해 코드 전체를 검색해야 함
❗ 새 기능 추가 시간이 점점 길어짐
```

**회피**: 처음부터 도메인별 모듈 경계를 설계하고, 모듈 간 통신은 정의된 인터페이스로만 허용. → [[12-architecture-patterns]] (모듈형 모놀리식)

---

### 2. Golden Hammer (만능 망치)

**정의**: 특정 기술/패턴에 익숙하거나 매력을 느껴, 적합성 여부와 관계없이 모든 문제에 그것을 적용하려는 경향.

**흔한 예시**:
```
❌ 트래픽이 적은 개인 프로젝트에 처음부터 마이크로서비스 도입
❌ 단순 CRUD 데이터에 NoSQL 도입 (관계형 데이터인데도)
❌ 모든 데이터 처리에 이벤트 소싱/CQRS 적용
❌ 익숙하다는 이유로 모든 문제를 같은 프레임워크로 해결
```

**회피**: "이 기술이 필요한가?"가 아니라 "이 문제에 가장 적합한 해결책은 무엇인가?"를 먼저 질문.

---

### 3. Stovepipe / Silo System

**정의**: 서로 통합되지 않은, 독립적으로 구축된 시스템/모듈들의 집합. 각 시스템이 데이터를 공유하거나 연계되지 않는다.

**신호**:
```
❗ 동일하거나 유사한 데이터가 여러 곳에 중복 저장
❗ 동일한 기능(인증, 로깅)이 여러 곳에 중복 구현
❗ 전체 데이터를 종합 조회하기 어려움
```

**회피**: 새 기능 추가 전 기존 데이터 모델/모듈과의 연계 지점을 먼저 검토. 공통 기능은 공유 모듈로 통합.

---

### 4. Magic Box (블랙박스 컴포넌트)

**정의**: 시스템의 특정 컴포넌트가 어떻게 동작하는지 아무도 제대로 이해하지 못하는 상태.

**발생 원인**:
- 복잡한 로직을 빠르게 구현하고 문서화를 생략
- 특정 라이브러리를 "일단 동작하니까" 사용하고 내부 원리를 이해하지 않음
- 설계 의도가 코드에만 존재하고 별도 문서로 남지 않음

> **개인 프로젝트에서 특히 위험**: 시간이 지나면 "과거의 나"가 "지금의 나"에게 Magic Box가 된다.

**회피**: 복잡한 로직·설계 결정은 반드시 ADR로 기록 ([[15-adr-and-c4-model]]). 핵심 데이터 흐름은 C4 다이어그램으로 시각화.

---

### 5. Reinventing the Wheel (바퀴 재발명)

**정의**: 이미 검증된 오픈소스 라이브러리/프레임워크가 존재하는데도, 처음부터 직접 만드는 것.

**흔한 예시**:
```
❌ 인증/인가 로직을 직접 구현 (JWT 라이브러리, OAuth Provider 이미 존재)
❌ ORM 대신 직접 SQL 빌더 구현
❌ 기존 UI 컴포넌트 라이브러리 기능을 처음부터 재구현
```

**문제점**:
- 핵심 비즈니스 로직이 아닌 "이미 풀린 문제"에 시간 소모
- 직접 구현한 코드는 검증된 라이브러리만큼의 보안성 갖추기 어려움

**회피**: 새 기능 설계 전 "이 문제를 해결하는 표준적인 라이브러리/서비스가 이미 있는가?" 먼저 확인.

---

### 6. God Object (전지전능 객체)

**정의**: 하나의 클래스/모듈/서비스가 너무 많은 책임을 가지고, 시스템의 거의 모든 로직과 데이터를 알고 있는 상태.

**신호**:
```
❗ 하나의 클래스/파일이 비정상적으로 많은 코드 라인 보유
   (일반적 기준: 메서드 30개 이상, 코드 400줄 이상이면 의심)
❗ 해당 클래스 수정 시 영향 범위를 예측하기 어려움
❗ 여러 도메인 개념(사용자, 주문, 결제 등)이 하나의 클래스에 혼재
```

**회피**: 단일 책임 원칙(SRP)에 따라 클래스/모듈 분리. 도메인별 모듈로 나누고 각 모듈이 자신의 책임 영역만 다루도록 설계.

---

## Details

### 안티패턴 체크리스트 (에이전트용)

설계 제안 전 반드시 점검:

```
□ 모듈/계층 경계가 명확하게 정의되어 있는가? (Big Ball of Mud 방지)
□ 현재 프로젝트 규모와 팀 크기에 적합한 패턴을 선택했는가? (Golden Hammer 방지)
□ 공통 기능(인증, 로깅)이 중복 구현되지 않는가? (Stovepipe 방지)
□ 핵심 설계 결정이 ADR로 기록되어 있는가? (Magic Box 방지)
□ 이미 검증된 라이브러리가 있는 기능을 직접 구현하려 하지 않는가? (Reinventing 방지)
□ 하나의 클래스/모듈이 너무 많은 책임을 지지 않는가? (God Object 방지)
```

### 안티패턴 발생 순서 (전형적인 패턴)

```
1. 빠른 MVP 출시를 위해 모듈 경계 없이 개발 시작
   → Big Ball of Mud 씨앗
   
2. 새 기능 추가 시 "이미 아는 방식"으로 무조건 적용
   → Golden Hammer
   
3. 각 기능이 독립적으로 구현, 통합 없이 누적
   → Stovepipe

4. 복잡해진 코드에 대한 문서화 생략
   → Magic Box

5. 기존 라이브러리를 쓰기 귀찮아서 직접 구현
   → Reinventing the Wheel

6. 모든 로직이 Service 클래스 하나에 집중
   → God Object
```
