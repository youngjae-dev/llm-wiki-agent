# 프로젝트 특성 분류 — 아키텍처 설계의 시작점

> 아키텍처 에이전트가 설계를 시작하기 전 반드시 수행하는 첫 번째 단계. 프로젝트의 "성격"을 6개 축으로 분류하여 이후 아키텍처 패턴 선택과 데이터 흐름 설계의 근거로 삼는다.

---

## Overview

좋은 아키텍처는 프로젝트의 성격을 정확히 이해하는 것에서 시작한다. 무작정 마이크로서비스를 도입하거나, 반대로 모든 것을 단일 서버에 몰아넣는 실수는 대부분 이 분류 단계를 건너뛴 결과다. 에이전트는 이 체크리스트를 채운 뒤 그 결과를 [[11-architecture-decision-guide]]의 입력으로 사용한다.

---

## Key Concepts

### 분류 체크리스트 (6개 축)

#### 1. 프로젝트 유형

| 유형 | 설명 | 예시 |
|---|---|---|
| 웹 서비스 | CRUD 중심, 사용자 인터랙션 多 | 쇼핑몰, 블로그 |
| 모바일 앱 백엔드 | API 중심, 오프라인 동기화 고려 | 배달앱 API |
| 데이터 파이프라인 | 정기적 ETL, 대량 데이터 변환 | 로그 집계, 배치 처리 |
| 실시간 스트리밍 | 이벤트 기반, 알림, 협업 | 채팅, 실시간 알림 |
| ML/AI 서비스 | 모델 서빙, 추론 파이프라인 | 추천 시스템, 이미지 분류 |
| 내부 도구 | 사용자 수 적음, 운영 부담 최소화 우선 | 사내 자동화 스크립트 |

#### 2. 트래픽/사용자 규모

- **동시 사용자 수** 추정 (개인 프로젝트: 수십~수백 명 vs 수만 명 이상)
- **트래픽 패턴**: 균일한가, 특정 시간에 몰리는가 (스파이크성)
- **향후 확장 가능성**: 지금은 작지만 커질 가능성

#### 3. 데이터 특성

| 항목 | 선택지 |
|---|---|
| 데이터 양 | 소규모(~수GB) / 대규모(수십GB+) |
| 데이터 구조 | 정형(테이블형) / 반정형(JSON) / 비정형(이미지, 텍스트) |
| 변경 빈도 | 읽기 多 / 쓰기 多 |
| 실시간성 | 즉시 반영 필요 / 약간의 지연 허용 |

#### 4. 일관성/신뢰성 요구

- **강한 일관성(Strong Consistency)** 필요: 결제, 재고, 금융
- **결과적 일관성(Eventual Consistency)** 허용: 피드, 알림, 통계
- 데이터 손실이 절대 허용되지 않는 영역 식별

#### 5. 팀/운영 제약

- 개발자 수 (1인 개발 vs 소규모 팀)
- 운영 부담 최소화 여부 (서버리스/관리형 서비스 선호도)
- 예산 제약 (무료/저비용 인프라)
- 배포 주기 (빠른 반복 vs 안정성 우선)

#### 6. 비기능 요구사항(NFR) 우선순위

다음 항목들에 대해 우선순위 매기기 (모두 동시 달성 불가 → 트레이드오프 필요):

1. 성능 (응답 속도)
2. 확장성 (스케일 아웃 용이성)
3. 가용성 (다운타임 최소화)
4. 보안
5. 유지보수성 / 개발 속도
6. 비용 효율성

## Details

### 에이전트 출력 형식

```
프로젝트 유형: 웹 서비스 (CRUD 중심)
예상 규모: 동시 사용자 수십~수백 명, 트래픽 균일
데이터 특성: 정형(RDB), 읽기/쓰기 균형, 즉시 반영 필요
일관성 요구: 강한 일관성 (주문/결제 포함)
운영 제약: 1인 개발, 무료 인프라, 빠른 반복 배포
NFR 우선순위: (1) 개발 속도 (2) 비용 효율 (3) 유지보수성
```

이 요약이 [[11-architecture-decision-guide]]의 입력으로 사용된다.

### 가장 흔한 실수

```
❌ 분류 없이 바로 아키텍처 선택
   → "요즘 마이크로서비스가 유행이니까" → 운영 지옥

❌ NFR 전체를 최우선으로 설정
   → 트레이드오프 없는 설계는 존재하지 않음

✅ 올바른 순서: 분류 → 의사결정 → 아키텍처 선택
```

## Examples / Code

### 분류 예시: 개인 블로그 서비스

```markdown
프로젝트 유형: 웹 서비스 (콘텐츠/읽기 중심)
예상 규모: 초기 수십 명, 향후 수천 명 가능
데이터 특성: 정형(게시글, 댓글), 읽기 90% / 쓰기 10%, 약간의 지연 허용
일관성 요구: 결과적 일관성 허용 (댓글 반영 1~2초 지연 OK)
운영 제약: 1인 개발, 최소 비용, 주 1회 배포
NFR 우선순위: (1) 개발 속도 (2) 비용 효율 (3) 유지보수성

→ 권장: 모듈형 모놀리식 + PostgreSQL + Redis 캐시
```

### 분류 예시: 실시간 협업 도구

```markdown
프로젝트 유형: 실시간 스트리밍 (WebSocket 기반)
예상 규모: 동시 수백 명, 스파이크 트래픽 있음
데이터 특성: 반정형(메시지, 이벤트), 쓰기 多, 즉시 반영 필수
일관성 요구: 결과적 일관성 허용 (메시지 순서 보장 필요)
운영 제약: 소규모 팀, 클라우드 인프라 사용 가능
NFR 우선순위: (1) 가용성 (2) 성능 (3) 확장성

→ 권장: 이벤트 드리븐 아키텍처 + Redis Pub/Sub + Message Queue
```
