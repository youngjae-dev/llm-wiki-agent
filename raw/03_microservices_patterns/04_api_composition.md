# API Composition 패턴

## 문제
Database per Service 패턴으로 인해 데이터가 여러 서비스/DB에 분산되면,
"여러 서비스의 데이터를 조합해야 하는 조회"를 하나의 DB Join으로 처리할 수 없다.
예: "주문 목록 + 각 주문의 상품 정보 + 배송 상태"를 한 화면에 보여줘야 하는 경우.

## 해결책
API Composer(별도 서비스 또는 API Gateway, 클라이언트)가 관련된 각 서비스에
개별적으로 요청을 보내고, 응답 결과를 메모리상에서 조합(Join)하여 클라이언트에 반환한다.

## 트레이드오프
- 장점: 서비스별 DB 독립성을 유지하면서 조합 조회 가능, 구현이 비교적 직관적
- 단점: 여러 서비스를 호출해야 하므로 지연 시간 증가, 호출 대상 서비스 중 하나가
  느리거나 실패하면 전체 응답에 영향 (병렬 호출 + 타임아웃/기본값 처리로 완화)
- 대량 데이터에 대한 정렬/페이징/필터링을 애플리케이션 레벨에서 구현해야 해서 비효율적일 수 있음
  → 이런 경우 CQRS(조회 전용 뷰 생성)가 더 적합할 수 있음

## 적용 기준
- 조합해야 하는 데이터의 양이 적고, 호출 대상 서비스 수가 많지 않을 때 적합
- 조합 로직이 복잡하거나 성능이 중요한 조회라면 CQRS의 별도 조회 모델을 고려
