# API Gateway 패턴

## 문제
클라이언트가 여러 마이크로서비스에 각각 직접 요청을 보내면,
클라이언트가 서비스의 위치/주소를 모두 알아야 하고, 인증/로깅/속도 제한 같은
공통 로직을 모든 서비스가 각자 구현해야 한다.

## 해결책
클라이언트의 모든 요청을 받는 단일 진입점(API Gateway)을 두고,
이 게이트웨이가 적절한 내부 서비스로 요청을 라우팅한다.
공통 관심사(인증, 로깅, Rate Limiting, 응답 조합)를 게이트웨이에서 처리한다.

## 장점
- 클라이언트 통합이 단순화됨 (엔드포인트 하나만 알면 됨)
- 인증/로깅/속도제한 등 공통 로직을 한 곳에서 관리
- 내부 서비스 구조가 바뀌어도 클라이언트에 영향 없음

## 단점
- 게이트웨이 자체가 단일 장애점(SPOF)이 될 수 있음 → 이중화 필요
- 게이트웨이에 너무 많은 로직이 몰리면 그 자체가 또 하나의 모놀리식이 될 위험

## 개인 프로젝트 적용 가이드
- 모듈형 모놀리식 단계에서는 별도 게이트웨이 없이, 애플리케이션 진입점 자체가
  이 역할(인증, 로깅 등 공통 미들웨어)을 한다고 보면 된다.
- 서비스를 여러 개로 분리하는 시점부터 명시적인 API Gateway 도입을 고려한다.
