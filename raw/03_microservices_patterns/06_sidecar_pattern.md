# Sidecar 패턴

## 문제
로깅, 모니터링, 인증, 네트워크 통신 처리 같은 공통 기능을 모든 서비스가
각자의 코드로 구현하면 중복이 발생하고, 언어/프레임워크가 다른 서비스마다
구현 방식이 달라진다.

## 해결책
메인 애플리케이션 컨테이너 옆에, 공통 기능을 담당하는 별도의 보조 프로세스(Sidecar)를
같은 호스트/Pod에 배치한다. 메인 애플리케이션은 이 보조 프로세스를 통해
로깅, 모니터링, 통신 등을 처리하며, 자신의 핵심 로직에만 집중한다.

- 관련 패턴: Ambassador (서비스 간 통신을 사이드카가 대리/추상화),
  Adapter (외부 시스템과의 인터페이스를 표준화)

## 트레이드오프
- 장점: 공통 기능을 언어/프레임워크에 무관하게 일관되게 적용 가능, 메인 애플리케이션 코드가 단순해짐
- 단점: 리소스 사용량 증가 (프로세스/컨테이너가 추가됨), 배포 단위가 복잡해짐

## 개인 프로젝트 적용 가이드
- 컨테이너 오케스트레이션(Kubernetes 등)을 사용하는 경우에 주로 의미가 있는 패턴이다.
- 단일 서버에 배포하는 소규모 프로젝트에서는 이 패턴을 직접 도입할 필요는 낮으며,
  대신 로깅/모니터링 라이브러리를 공통 모듈로 분리하는 것으로 충분한 경우가 많다.
