# Database per Service

## 문제
마이크로서비스 아키텍처에서 여러 서비스가 하나의 DB를 공유하면,
서비스 간 결합도가 높아져 독립적인 배포/스케일링이 어려워진다.
(이를 "Shared Database" 안티패턴이라 부른다)

## 해결책
각 서비스는 자신만의 데이터 저장소를 가지며, 다른 서비스는 그 데이터에
직접 접근할 수 없고 반드시 해당 서비스의 API를 통해서만 접근한다.

- 서비스별로 데이터 특성에 맞는 DB를 선택할 수 있음 (폴리글랏 퍼시스턴스)
  - 예: 상품 카탈로그 → 문서형 DB (유연한 스키마)
  - 주문/결제 → RDBMS (트랜잭션 보장)
  - 분석 → 데이터 웨어하우스

## 트레이드오프
- 장점: 서비스 간 결합도 감소, 서비스별 최적 DB 선택 가능, 독립 배포 용이
- 단점: 여러 서비스의 데이터를 묶어 조회하는 작업(Join)이 애플리케이션 레벨로 이동 →
  API Composition 또는 CQRS 패턴 필요. 데이터 일관성 유지가 어려워짐 → Saga 패턴 필요

## 개인 프로젝트 적용 가이드
- 모듈형 모놀리식 단계에서는 "하나의 DB, 모듈별 스키마 분리"로 시작하는 것이 현실적이다.
- 실제로 서비스를 물리적으로 분리하는 시점에 이 패턴을 적용한다.
