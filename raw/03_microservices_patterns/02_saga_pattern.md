# Saga 패턴

## 문제
마이크로서비스마다 별도의 DB를 가지면(Database per Service), 여러 서비스에
걸친 작업을 하나의 트랜잭션으로 묶을 수 없다 (분산 트랜잭션의 어려움).
예: "주문 생성 → 결제 처리 → 재고 차감"이 서로 다른 서비스/DB에 걸쳐 있는 경우.

## 해결책
하나의 비즈니스 트랜잭션을, 각 서비스가 처리하는 일련의 "로컬 트랜잭션"으로 나누고,
각 단계가 끝나면 다음 단계를 트리거하는 이벤트/메시지를 발행한다.
중간에 실패하면, 이전 단계들을 취소하는 "보상 트랜잭션(Compensating Transaction)"을 실행한다.

### 두 가지 구현 방식
- Choreography(코레오그래피): 각 서비스가 이벤트를 구독하고 자율적으로 다음 행동을 결정
  (중앙 제어자 없음, 단순한 흐름에 적합)
- Orchestration(오케스트레이션): 별도의 오케스트레이터가 각 단계를 순서대로 호출하고
  실패 시 보상 트랜잭션을 지시 (복잡한 흐름의 가시성/관리에 유리)

## 트레이드오프
- 장점: 서비스별 DB 독립성을 유지하면서도 여러 서비스에 걸친 작업을 처리 가능
- 단점: ACID 트랜잭션처럼 즉각적인 롤백이 아니라 "보상" 로직을 직접 구현해야 함,
  중간 상태가 일시적으로 노출될 수 있음 (결과적 일관성)

## 적용 기준
- 여러 서비스에 걸친 쓰기 작업이 있고, 그 작업이 실패할 경우의 복구 로직이
  비즈니스적으로 중요한 경우에 적용한다.
- 단일 DB 트랜잭션으로 처리 가능한 범위라면(모놀리식 내부) Saga가 필요 없다.
