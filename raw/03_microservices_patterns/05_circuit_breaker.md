# Circuit Breaker 패턴

## 문제
한 서비스가 의존하는 외부 서비스/API가 느려지거나 응답하지 않을 때,
호출하는 쪽이 계속 요청을 보내고 응답을 기다리면, 자원(스레드, 커넥션)이
모두 그 대기 상태에 묶여 호출하는 서비스 자체도 다운될 수 있다 (장애 전파).

## 해결책
호출 실패율을 모니터링하다가, 일정 기준을 넘으면 "회로(Circuit)를 차단(Open)"하여
이후 요청을 즉시 실패 처리하거나 대체 응답(Fallback)을 반환한다.
일정 시간 후 일부 요청만 통과시켜(Half-Open) 대상 서비스가 복구되었는지 확인하고,
정상이면 회로를 다시 닫는다(Closed).

## 상태 전이
- Closed: 정상 상태, 요청을 그대로 전달
- Open: 장애 감지, 요청을 즉시 실패/Fallback 처리 (대상 서비스 호출 안 함)
- Half-Open: 일부 요청만 통과시켜 복구 여부 확인

## 트레이드오프
- 장점: 장애 전파 방지, 응답 불가 상태에서 빠른 실패(Fail Fast)로 사용자 경험 개선
- 단점: Fallback 로직을 별도로 설계해야 함, 임계값(threshold) 튜닝 필요

## 적용 기준
- 외부 API, 다른 마이크로서비스 등 "내가 직접 제어할 수 없는 의존성"을 호출하는
  모든 지점에 기본적으로 고려할 만한 패턴
- 개인 프로젝트에서도 외부 결제/알림 API 연동 시 적용하면 안정성이 크게 향상됨
