# 안티패턴: Reinventing the Wheel

## 정의
이미 검증된 오픈소스 라이브러리/프레임워크/도구가 존재하는데도,
이를 사용하지 않고 처음부터 직접 만드는 것.

## 흔한 예시
- 인증/인가 로직을 직접 구현 (JWT 라이브러리, OAuth 제공자 등이 이미 존재)
- ORM 대신 직접 SQL 빌더/쿼리 추상화 계층을 구현
- 이미 존재하는 UI 컴포넌트 라이브러리의 기능을 처음부터 다시 구현

## 문제점
- 시간/리소스가 핵심 비즈니스 로직이 아닌 "이미 풀린 문제"에 소모됨
- 직접 구현한 코드는 검증된 라이브러리만큼의 안정성/보안성을 갖추기 어려움
  (특히 인증/보안 영역에서 위험)
- 유지보수 부담이 전적으로 본인에게 귀속됨

## 회피 방법
- 새로운 기능을 설계하기 전, "이 문제를 해결하는 표준적인 라이브러리/서비스가
  이미 있는가?"를 먼저 확인
- 직접 구현을 선택하는 경우, 그 이유(예: 라이브러리가 요구사항을 만족하지 못함,
  의존성을 최소화해야 함)를 ADR에 명시

## agent 설계 시 적용
- agent는 특정 기능(인증, 결제, 파일 업로드, 이미지 처리 등)을 설계할 때,
  먼저 해당 영역의 표준적인 라이브러리/서비스 옵션을 제시하고,
  "직접 구현"은 명확한 이유가 있을 때만 옵션으로 제안한다.
