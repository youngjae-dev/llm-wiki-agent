# 안티패턴: God Object

## 정의
하나의 클래스/모듈/서비스가 너무 많은 책임을 가지고, 시스템의 거의 모든
로직과 데이터를 알고 있는 상태. "모든 것을 알고 모든 것을 하는" 객체.

## 발생 원인
- 새로운 로직을 추가할 때, 적절한 위치를 찾기보다 기존의 "큰" 클래스에 계속 추가
- 책임 분리(Separation of Concerns) 원칙이 처음부터 적용되지 않음

## 신호 (Signs)
- 하나의 클래스/파일의 코드 라인 수가 다른 것들에 비해 비정상적으로 많음
  (일반적인 기준: 메서드 30개 이상, 코드 라인 400줄 이상이면 "Large Class" 의심)
- 해당 클래스를 수정할 때 영향 범위를 예측하기 어려움
- 여러 도메인 개념(사용자, 주문, 결제 등)이 하나의 클래스 안에 섞여 있음

## 회피 방법
- 단일 책임 원칙(SRP)에 따라 클래스/모듈을 분리
- 도메인별로 모듈을 나누고(02_architecture_patterns/01_layered_modular_monolith.md),
  각 모듈이 자신의 책임 영역만 다루도록 설계
- 새 기능 추가 시, "이 로직이 속한 도메인이 무엇인가"를 먼저 판단하고 해당 모듈에 배치

## agent 설계 시 적용
- 초기 설계 단계에서 도메인/모듈을 명확히 나누어 두면, 이후 기능이 늘어나도
  특정 모듈로 책임이 과도하게 집중되는 것을 예방할 수 있다.
- agent는 설계 결과에 "각 모듈이 담당하는 책임의 범위"를 명시적으로 정의해야 한다.
