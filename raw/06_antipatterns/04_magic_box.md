# 안티패턴: Magic Box

## 정의
시스템의 특정 컴포넌트가 어떻게 동작하는지 아무도(혹은 극히 일부만) 제대로
이해하지 못하는 상태. 내부 로직이 복잡하고 문서화가 부족하여 "마법의 상자"처럼
취급된다.

## 발생 원인
- 복잡한 로직을 빠르게 구현하고 문서화/주석을 남기지 않음
- 특정 라이브러리/서비스를 "일단 동작하니까" 사용하고, 그 동작 원리를 이해하지 않음
- 설계 의도(왜 이렇게 만들었는지)가 코드에만 존재하고 별도 문서로 남지 않음

## 문제점
- 해당 컴포넌트에 문제가 생기면 수정이 매우 어려움 (사이드 이펙트 예측 불가)
- 개발자가 1인이라도, 시간이 지나면 "과거의 나"가 "지금의 나"에게 Magic Box가 됨
  (개인 프로젝트에서도 매우 흔하게 발생)

## 회피 방법
- 복잡한 로직/설계 결정은 반드시 ADR로 남김 (왜 이렇게 만들었는지)
- 핵심 데이터 흐름과 모듈 간 의존 관계를 C4 다이어그램으로 시각화하여 문서화
- 코드 자체보다 "왜(Why)"에 대한 기록을 우선시

## agent 설계 시 적용
- agent가 생성하는 모든 설계 산출물(ADR, 다이어그램)은 향후 "내가 왜 이렇게
  설계했는지"를 다시 확인할 수 있는 자료가 되어야 한다. 따라서 모든 주요 결정에는
  배경(Context)과 근거(Argument)를 반드시 포함시킨다.
