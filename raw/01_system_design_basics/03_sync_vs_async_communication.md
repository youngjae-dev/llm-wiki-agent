# 동기 vs 비동기 통신

## 동기 통신 (Synchronous)
- 예: REST API, gRPC
- 동작: 요청을 보낸 쪽이 응답이 올 때까지 대기
- 장점: 흐름이 직관적, 디버깅 용이, 즉시 결과 확인 가능
- 단점: 호출 대상이 느리거나 다운되면 호출자도 영향을 받음 (강한 결합)

## 비동기 통신 (Asynchronous)
- 예: 메시지 큐(RabbitMQ, Kafka), 이벤트 버스, Pub/Sub
- 동작: 요청을 보낸 쪽은 결과를 기다리지 않고 다음 작업을 계속함. 별도의 워커/구독자가 처리
- 장점: 컴포넌트 간 결합도 감소, 한쪽 장애가 다른 쪽에 즉시 전파되지 않음, 트래픽 버퍼링 가능
- 단점: 전체 흐름 추적이 어려움(분산 트레이싱 필요), 즉시 응답이 필요한 작업에는 부적합

## 선택 기준
| 상황 | 권장 방식 |
|---|---|
| 사용자가 결과를 즉시 확인해야 함 (로그인, 조회) | 동기 |
| 결과가 즉시 필요 없는 후속 작업 (이메일 발송, 썸네일 생성, 알림) | 비동기 |
| 여러 서비스에 동일 이벤트를 전파해야 함 | 비동기 (Pub/Sub) |
| 처리 순서가 보장되어야 하고 트랜잭션 일관성이 중요함 | 동기 + 트랜잭션, 또는 Saga(03_microservices_patterns/02_saga_pattern.md 참고) |

## 데이터 흐름 설계 시 체크포인트
- 사용자 요청 경로 중 "즉시 응답해야 하는 부분"과 "백그라운드로 미뤄도 되는 부분"을 분리한다.
- 비동기로 분리한 작업은 실패 시 재시도 정책(retry, dead-letter queue)을 함께 설계한다.
- 과도한 동기 호출 체인(A → B → C → D)은 장애 전파 위험이 크므로,
  필요 이상으로 동기 의존을 늘리지 않도록 주의한다 (안티패턴: Overuse of Synchronous Communication).
