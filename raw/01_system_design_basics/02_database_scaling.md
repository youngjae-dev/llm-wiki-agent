# 데이터베이스 스케일링 전략

## 수직 확장 (Scale Up)
- 단일 서버의 CPU/메모리/디스크 사양을 높이는 방식
- 장점: 구조 변경 없이 적용 가능, 단순함
- 단점: 물리적 한계 존재, 비용 대비 효율이 점점 낮아짐, 단일 장애점(SPOF) 위험

## 수평 확장 (Scale Out)

### 리플리케이션 (Replication)
- 동일한 데이터를 여러 서버(레플리카)에 복제
- Master-Replica 구조: 쓰기는 Master, 읍기는 Replica로 분산
- 장점: 읍 성능 향상, 장애 시 페일오버 가능
- 단점: 복제 지연(Replication Lag)으로 인한 일시적 데이터 불일치 가능

### 샤딩 (Sharding / Partitioning)
- 데이터를 여러 서버(샤드)에 분할 저장
- 분할 기준: 특정 키(사용자 ID, 지역 등)의 해시 또는 범위 기준
- 장점: 쓰기/읍 모두 수평 확장 가능
- 단점: 샤드 간 JOIN 어려움, 리샤딩(샤드 재분배) 비용이 큼, 운영 복잡도 급증

## 일관성 모델
- 강한 일관성(Strong Consistency): 모든 노드가 항상 동일한 최신 데이터를 본다. 구현 비용이 크고 지연 증가
- 결과적 일관성(Eventual Consistency): 일정 시간이 지나면 모든 노드가 동기화됨. 가용성과 성능에 유리

## 개인 프로젝트에 대한 가이드
- 대부분의 개인/소규모 프로젝트는 단일 DB 인스턴스로 충분하다.
- 읍 부하가 문제라면 먼저 캐싱(01_caching_strategies.md)을 적용하고,
  그래도 부족할 때 리플리케이션을 고려한다.
- 샤딩은 데이터/트래픽 규모가 매우 커진 이후에 고려할 단계로,
  초기 설계에 무리하게 포함시키지 않는다 (과설계 주의).
