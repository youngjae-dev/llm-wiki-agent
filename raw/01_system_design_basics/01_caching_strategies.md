# 캐싱 전략

## 핵심 개념
캐싱은 자주 사용되는 데이터를 더 빠른 저장소(메모리 등)에 임시 보관하여
응답 속도를 높이고 원본 데이터 소스(DB, API)의 부하를 줄이는 기법이다.

## 주요 패턴

### Cache-Aside (Lazy Loading)
- 동작: 애플리케이션이 먼저 캐시를 조회 → 없으면(Cache Miss) DB에서 읍어와 캐시에 저장 → 다음 요청부터는 캐시에서 응답
- 장점: 필요한 데이터만 캐시에 적재됨, 구현이 단순함
- 단점: 첫 요청은 항상 느림(Cache Miss), 캐시와 DB 간 데이터 불일치 가능성

### Write-Through
- 동작: 데이터를 쓸 때 캐시와 DB에 동시에 기록
- 장점: 캐시와 DB의 데이터가 항상 최신 상태로 동기화됨
- 단점: 쓰기 작업의 지연 시간이 증가함

### Write-Back (Write-Behind)
- 동작: 캐시에 먼저 쓰고, 일정 주기로 DB에 비동기 반영
- 장점: 쓰기 성능이 매우 빠름
- 단점: 캐시 장애 시 데이터 손실 위험이 있음

## 캐시 무효화(Invalidation) 전략
- TTL(Time-To-Live) 기반: 일정 시간이 지나면 자동 만료
- 이벤트 기반: 원본 데이터가 변경되면 명시적으로 캐시 삭제/갱신
- 버전 기반: 데이터에 버전을 부여하고, 버전이 바뀌면 캐시 키도 변경

## 트레이드오프
- 장점: 응답 속도 개선, DB/API 부하 감소, 비용 절감
- 단점: 캐시-원본 데이터 간 일관성 문제(Cache Coherency), 무효화 로직의 복잡도 증가

## 적용 판단 기준
- 동일한 데이터에 대한 읍 요청이 빈번한가
- 데이터가 자주 바뀌지 않거나, 약간의 지연된 데이터(Stale Data)가 허용되는가
- 위 조건을 만족할수록 캐싱 도입 효과가 크다.
