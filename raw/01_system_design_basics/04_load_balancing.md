# 로드 밸런싱

## 핵심 개념
하나의 서버로 처리하기 어려운 트래픽을 여러 서버 인스턴스에 분산시키는 기법.
가용성(한 서버 장애 시 다른 서버가 대신 처리)과 확장성을 모두 향상시킨다.

## 주요 분산 알고리즘
- Round Robin: 요청을 순서대로 각 서버에 균등 분배. 서버 성능이 비슷할 때 적합
- Least Connections: 현재 처리 중인 연결 수가 가장 적은 서버로 분배. 요청 처리 시간이 불균일할 때 적합
- IP Hash / Consistent Hashing: 특정 클라이언트의 요청을 항상 같은 서버로 보냄. 세션 유지가 필요할 때 활용

## 적용 위치
- 클라이언트 ↔ 서버 사이 (L7 로드밸런서, API Gateway 단)
- 서버 ↔ DB 사이 (읍 전용 레플리카로 분산)

## 개인 프로젝트 관점
- 트래픽이 적은 초기 단계에서는 로드밸런서가 필수는 아니다.
- 다만 배포 플랫폼(예: 클라우드 PaaS)이 기본 제공하는 로드밸런싱은
  무중단 배포, 헬스체크 용도로도 유용하므로 활용을 고려한다.
- 트래픽이 늘어나는 시점에 인스턴스를 늘리고 로드밸런서를 앞에 두는 구조로
  전환하기 쉽도록, 애플리케이션을 "상태 비저장(Stateless)"으로 설계하는 것이 핵심이다.
  (세션 정보는 서버 메모리가 아닌 외부 저장소(Redis 등)에 둘 것)
