# 마이크로서비스 아키텍처

## 핵심 개념
하나의 큰 애플리케이션을 비즈니스 기능 단위의 작은 서비스들로 분리하고,
각 서비스가 독립적으로 개발/배포/확장될 수 있도록 하는 아키텍처 스타일.
각 서비스는 보통 자신만의 데이터 저장소를 가진다 (Database per Service, 03_microservices_patterns 참고).

## 장점
- 서비스별 독립 배포 및 확장 (트래픽이 많은 서비스만 스케일 아웃 가능)
- 기술 스택을 서비스별로 다르게 선택 가능 (Polyglot)
- 장애 격리: 한 서비스의 장애가 전체 시스템 다운으로 이어질 가능성 감소

## 단점 / 비용
- 서비스 간 통신(네트워크 호출)이 늘어나 지연 및 장애 가능성 증가
- 여러 서비스의 데이터를 조합한 조회(Join)가 어려워짐 → API Composition, CQRS 같은
  별도 패턴 필요
- 분산 트랜잭션 처리가 복잡함 → Saga 패턴 필요
- 운영 복잡도 급증: 서비스 디스커버리, 모니터링, 배포 파이프라인을 각 서비스마다 관리해야 함

## 모놀리식과의 관계
모놀리식 아키텍처는 마이크로서비스의 대안이며, 다음 패턴들은 마이크로서비스를
도입했을 때 발생하는 문제들을 해결하기 위한 보조 패턴들이다:
- 통신: Messaging, Remote Procedure Invocation
- 데이터 조회: API Composition, CQRS
- 트랜잭션: Saga, Transactional Outbox

## 도입 판단 기준
- 팀(또는 향후 팀)이 서비스별로 독립적으로 작업할 필요가 있는가
- 특정 기능의 트래픽/리소스 요구량이 다른 기능과 크게 다른가
- 위 조건이 약하다면, 모듈형 모놀리식으로 시작하고 경계를 명확히 유지하는 것이
  장기적으로도 더 안전한 선택이다.
