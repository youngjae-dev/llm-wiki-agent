# 이벤트 드리븐 아키텍처 (Event-Driven Architecture)

## 핵심 개념
시스템의 컴포넌트들이 "이벤트(상태 변화에 대한 사실의 기록)"를 발행(publish)하고,
관심 있는 다른 컴포넌트들이 이를 구독(subscribe)하여 반응하는 구조.
컴포넌트 간 직접 호출 대신 이벤트 버스/메시지 브로커를 통해 통신한다.

## 구성 요소
- Producer (이벤트 발행자): 상태 변화가 발생한 쪽 (예: "주문이 생성됨")
- Event Broker: 이벤트를 전달하는 매개체 (예: Kafka, RabbitMQ, AWS SNS/SQS)
- Consumer (이벤트 구독자): 이벤트에 반응하여 후속 작업 수행 (예: 알림 발송, 재고 차감)

## 장점
- 컴포넌트 간 느슨한 결합: Producer는 누가 이벤트를 소비하는지 몰라도 됨
- 새로운 기능 추가 시 기존 코드를 수정하지 않고 새 Consumer만 추가 가능
- 트래픽 버퍼링: 처리 속도가 다른 컴포넌트 간 속도 차이를 흡수

## 단점
- 전체 데이터 흐름을 한눈에 파악하기 어려움 (이벤트 추적/로깅 인프라 필요)
- 이벤트 순서 보장, 중복 처리(at-least-once delivery) 등 추가 고려사항 발생
- 디버깅이 동기 시스템보다 복잡함

## 데이터 흐름 설계 시 고려사항
- 어떤 이벤트가 "사실의 기록(Fact)"으로서 발행되어야 하는지 명확히 정의
- 이벤트 스키마(필드 구조)를 버전 관리하여, Consumer가 늘어나도 호환성을 유지
- Consumer 실패 시 재처리를 위한 Dead Letter Queue 설계

## 개인 프로젝트 적용 가이드
- 모든 통신을 이벤트 기반으로 만들 필요는 없다.
- "즉시 응답이 필요 없는 후속 처리"(알림, 통계 집계, 로그 적재) 영역에 한정해서
  부분적으로 도입하는 것이 현실적이다 (모듈형 모놀리식 내부에서도 in-process 이벤트로 시작 가능).
