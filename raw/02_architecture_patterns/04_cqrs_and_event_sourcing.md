# CQRS와 이벤트 소싱 (Event Sourcing)

## CQRS (Command Query Responsibility Segregation)
"명령(쓰기)"과 "조회(읍기)"의 책임을 분리하는 패턴.

- Command 모델: 데이터를 변경하는 작업 전용. 비즈니스 규칙 검증, 트랜잭션 처리
- Query 모델: 데이터를 조회하는 작업 전용. 화면에 맞게 최적화된 형태로 데이터 제공
- 두 모델은 같은 DB를 쓸 수도 있고, 완전히 분리된 저장소(예: 쓰기는 RDBMS, 조회는 캐시/검색엔진)를 쓸 수도 있음

### 장점
- 조회 성능을 별도로 최적화 가능 (복잡한 Join을 미리 계산해 둔 뷰 테이블 등)
- 쓰기 모델과 조회 모델이 독립적으로 확장 가능

### 단점
- 두 모델 간 데이터 동기화 지연 발생 가능 (결과적 일관성)
- 구조가 단순한 CRUD 대비 복잡도가 크게 증가

### 적용 기준
- 읍 패턴이 매우 복잡하거나(여러 화면에서 다른 형태로 같은 데이터를 보여줘야 함),
  읍/쓰기 부하 특성이 크게 다를 때만 도입을 고려한다.
- 단순 CRUD 위주의 개인 프로젝트에는 보통 과한 설계이다.

## 이벤트 소싱 (Event Sourcing)
데이터의 "현재 상태"를 직접 저장하는 대신, 상태를 변화시킨 "이벤트(변경 이력)"들을
순서대로 저장하고, 현재 상태는 이벤트들을 재생(replay)하여 계산하는 방식.

- 예: "계좌 잔액 = 1000원"을 직접 저장하지 않고,
  "입금 1000원", "출금 200원" 같은 이벤트들의 합으로 잔액을 계산

### 장점
- 모든 변경 이력이 보존되어 감사(audit)/디버깅에 유리
- 특정 시점의 상태를 재구성할 수 있음
- CQRS와 결합하면 다양한 조회용 뷰(Projection)를 이벤트로부터 생성 가능

### 단점
- 구현 복잡도가 매우 높음
- 이벤트 스키마 변경 시 과거 이벤트와의 호환성 관리 필요
- 현재 상태 조회를 위해 이벤트를 재생하는 비용 → 스냅샷(Snapshot) 전략 필요

### 적용 기준
- 변경 이력 자체가 비즈니스적으로 중요한 도메인(금융 거래, 재고 변동 추적 등)에서만 고려
- 일반적인 개인 프로젝트에서는 거의 필요하지 않으며, 도입 시 "Golden Hammer"
  안티패턴이 되기 쉬우므로 신중히 판단할 것
