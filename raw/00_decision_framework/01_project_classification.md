# 프로젝트 특성 분류 체크리스트

설계를 시작하기 전, agent는 아래 항목들을 파악해서 프로젝트의 "성격"을 먼저 정의해야 한다.
이 분류 결과가 이후 아키텍처 패턴 선택, 데이터 흐름 설계의 근거가 된다.

## 1. 프로젝트 유형
- 웹 서비스 (CRUD 중심, 사용자 인터랙션 多)
- 모바일 앱 백엔드 (API 중심, 오프라인 동기화 고려 필요할 수 있음)
- 데이터 파이프라인 / 배치 처리 (정기적 ETL, 대량 데이터 변환)
- 실시간 스트리밍 / 이벤트 기반 시스템 (로그 처리, 알림, 협업툴)
- ML/AI 서비스 (모델 서빙, 추론 파이프라인, 학습 데이터 관리)
- 내부 도구 / 자동화 스크립트 (사용자 수 적음, 운영 부담 최소화 우선)

## 2. 트래픽 / 사용자 규모
- 동시 사용자 수 추정 (개인 프로젝트: 수십~수백 명 vs 수만 명 이상)
- 트래픽 패턴: 균일한가, 특정 시간에 몰리는가 (스파이크성)
- 향후 확장 가능성: 지금은 작지만 커질 가능성이 있는가

## 3. 데이터 특성
- 데이터 양: 소규모(수백MB~수GB) vs 대규모(수십GB 이상)
- 데이터 구조: 정형(테이블형) / 반정형(JSON) / 비정형(이미지, 텍스트, 로그)
- 데이터 변경 빈도: 읍/쓰기 비율 (읍이 많은가, 쓰기가 많은가)
- 실시간성 요구: 즉시 반영되어야 하는가, 약간의 지연이 허용되는가

## 4. 일관성/신뢰성 요구
- 강한 일관성(Strong Consistency)이 필요한 도메인인가 (결제, 재고 등)
- 결과적 일관성(Eventual Consistency)으로 충분한 도메인인가 (피드, 알림 등)
- 데이터 손실이 절대 허용되지 않는 영역이 있는가

## 5. 팀/운영 제약
- 개발자 수 (1인 개발인지, 소규모 팀인지)
- 운영 부담을 최소화해야 하는가 (서버리스/관리형 서비스 선호)
- 예산 제약 (무료/저비용 인프라 위주인가)
- 배포 주기: 빠른 반복 배포가 중요한가, 안정성이 더 중요한가

## 6. 비기능 요구사항(NFR) 우선순위
다음 항목들에 대해 우선순위를 매긴다 (보통 모두 만족시킬 수 없으므로 트레이드오프 필요):
- 성능(응답 속도)
- 확장성(스케일 아웃 용이성)
- 가용성(다운타임 최소화)
- 보안
- 유지보수성 / 개발 속도
- 비용 효율성

## 출력 형식
agent는 이 체크리스트를 채운 뒤, 다음 형태로 요약을 출력한다:

```
프로젝트 유형: ...
예상 규모: ...
데이터 특성: ...
일관성 요구: ...
운영 제약: ...
NFR 우선순위 (1~3위): ...
```

이 요약이 다음 단계인 02_architecture_style_decision_guide.md 의 입력으로 사용된다.
