# 아키텍처 스타일 결정 가이드

`01_project_classification.md`에서 도출한 프로젝트 특성을 바탕으로,
어떤 아키텍처 스타일과 데이터 흐름 방식이 적합한지 판단하는 매핑 가이드.

## 1. 프로젝트 규모 → 아키텍처 스타일

| 조건 | 추천 스타일 | 이유 |
|---|---|---|
| 1인 개발, MVP, 사용자 수 적음 | 모듈형 모놀리식(Modular Monolith) | 배포/운영 단순, 개발 속도 빠름, 추후 분리 용이 |
| 기능별 독립 배포/스케일링 필요, 팀 분리 가능 | 마이크로서비스 | 서비스별 독립 확장, 단 운영 복잡도 증가 |
| 이벤트 발생량이 많고 비동기 처리가 핵심 | 이벤트 드리븐 아키텍처 | 느슨한 결합, 비동기 처리에 유리 |
| 서버 운영 부담을 최소화하고 싶음 | 서버리스(Serverless) | 트래픽 기반 과금, 운영 자동화, 단 콜드스타트/벤더 종속 고려 |

> 개인 프로젝트는 대부분 "모듈형 모놀리식"으로 시작해서, 특정 모듈의 부하가 커지면
> 그 부분만 분리하는 점진적 접근이 안전하다. (안티패턴: 처음부터 마이크로서비스로 시작하는 "Golden Hammer" 참고)

## 2. 읍/쓰기 비율 → 데이터 처리 패턴

- 읍이 압도적으로 많은 경우 (피드, 콘텐츠 사이트):
  - 캐싱 레이어 도입 (Cache-Aside 패턴)
  - 읍 전용 복제본(Read Replica) 고려
  - CQRS로 읍/쓰기 모델 분리 고려 (단, 소규모 프로젝트엔 과할 수 있음)
- 쓰기가 많고 처리 순서가 중요한 경우 (로그 수집, 주문 처리):
  - 메시지 큐를 통한 비동기 처리 (Producer-Consumer)
  - 쓰기 자체는 빠르게 받고, 처리는 백그라운드 워커로 분리

## 3. 일관성 요구 → 통신/트랜잭션 방식

| 일관성 요구 | 통신 방식 | 트랜잭션 처리 |
|---|---|---|
| 강한 일관성 필요 (결제, 재고) | 동기(REST/RPC) + 단일 DB 트랜잭션 | ACID 트랜잭션, 분산이 필요하면 Saga 패턴 |
| 결과적 일관성 허용 (알림, 통계, 추천) | 비동기(메시지 큐, 이벤트) | 이벤트 기반 업데이트, 재시도/보정 로직 |

## 4. 데이터 구조 → DB 선택

- 정형 데이터 + 관계/트랜잭션 중요 → RDBMS (PostgreSQL, MySQL)
- 스키마가 자주 바뀌거나 JSON 형태 데이터 → 문서형 DB (MongoDB)
- 단순 키-값 조회, 캐싱 → KV 스토어 (Redis)
- 관계가 복잡하고 탐색이 중요 (추천, 소셜 그래프) → 그래프 DB (Neo4j)
- 대량 로그/시계열 데이터 분석 → 컬럼형/시계열 DB (TimescaleDB, ClickHouse)

> 여러 DB를 섞는 "폴리글랏 퍼시스턴스"는 강력하지만, 개인 프로젝트에서는
> 운영 복잡도를 고려해 1~2개 DB로 시작하고 필요할 때 추가하는 것을 권장.

## 5. 데이터 흐름 설계 체크리스트

agent는 아키텍처를 확정하기 전, 다음 데이터 흐름을 명시적으로 그려야 한다:
1. 사용자/클라이언트 요청이 진입하는 지점 (API Gateway / 단일 엔드포인트)
2. 요청이 동기로 처리되는 부분과 비동기로 위임되는 부분의 경계
3. 데이터가 저장되는 위치와 그 데이터를 읍는 다른 컴포넌트
4. 캐시가 적용되는 지점과 무효화(invalidation) 시점
5. 외부 서비스/API와의 연동 지점 및 장애 시 대응(타임아웃, 재시도, 서킷브레이커)

## 6. 최종 출력 형식

agent는 설계 결과를 `04_adr_templates`의 형식에 맞춰 ADR(Architecture Decision Record)로 정리하고,
`05_c4_model`의 표기법으로 Context/Container 수준 다이어그램을 함께 제시한다.
