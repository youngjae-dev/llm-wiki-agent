# 지식 도메인 정의 문서

## 도메인: 개인 프로젝트 시스템 아키텍처 설계

---

## 1. 도메인 선택 이유

개인 프로젝트를 시작할 때 가장 먼저 맞닥뜨리는 문제는 "어떻게 설계해야 하는가?"다. 기능 구현은 검색으로 해결할 수 있지만, **아키텍처 결정**은 프로젝트 초기에 내려야 하며 나중에 변경하기 매우 어렵다. 이 결정을 잘못 내리면:

- 트래픽이 없는 서비스에 마이크로서비스를 도입해 운영 비용이 폭증
- 읽기 위주 서비스에 복잡한 CQRS를 적용해 불필요한 복잡도 발생
- 단순 CRUD에 Event Sourcing을 적용해 유지보수가 불가능해짐

ArchWiki는 이 문제를 해결하기 위한 **아키텍처 설계 보조 에이전트**의 지식 베이스다.

## 3. 에이전트의 역할

ArchWiki Agent는 다음 질문에 답할 수 있어야 한다:

1. "이 프로젝트에 어떤 아키텍처 패턴이 적합한가?"
2. "읽기가 많고 쓰기가 적은 경우 어떻게 설계하는가?"
3. "마이크로서비스와 모듈형 모놀리식 중 어느 것이 나에게 맞는가?"
4. "캐싱 전략을 어떻게 선택하는가?"
5. "내 설계에 안티패턴이 있는가?"

---

## 4. 도메인 경계

**포함하는 것**:
- 아키텍처 패턴 선택 기준
- 데이터 흐름 설계 (캐싱, DB 스케일링, 통신 방식)
- 설계 결정 기록 방법 (ADR)
- 시각화 방법 (C4 모델)
- 설계 실수 방지 (안티패턴)

**포함하지 않는 것**:
- 특정 언어/프레임워크 구현 코드
- 인프라·클라우드 서비스 세부 설정
- 보안, 인증 설계 (별도 도메인)
- 비즈니스 로직 설계
